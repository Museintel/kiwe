# Seam vocabulary contract

Seam is the page and section framework that sits underneath Kiwe's AppShell. Kiwe/DSA owns the dock, screens, cart, checkout, profile, search, AI, PhoneKey, PWA, and runtime lifecycle. Seam owns the normal webpage vocabulary: what a block is, where it flows, how intense it should feel, what tone it uses, and what visual state it is in.

The rule is simple:

- Attributes are the canonical brain.
- Classes are the adoption bridge for Bricks, WordPress builders, junior developers, and AI-generated HTML.
- Both doors must produce the same semantic meaning. Seam roles are headless by default: they name the element without adding generic card/button/modal visuals.

Example:

```html
<section data-role="section" data-flow="stack" data-scene="elevated">
```

and:

```html
<section class="seam-section seam-stack seam-scene-elevated">
```

should mean the same thing.

## The seven questions

1. What is it? `data-role` / `.seam-*`
2. Where/how does it flow? `data-flow` / `.seam-stack`, `.seam-grid`, etc.
3. What scene is it in? `data-scene` / `.seam-scene-*`
4. What tone does it use? `data-tone` / `.seam-tone-*`
5. What state is it in? `data-state` / `.seam-is-*`
6. What explicit shape utility does it use, if any? `data-shape` / `.seam-shape-*`
7. Is it presentation or authority? Seam presents; Kiwe/Woo/WordPress/Bricks own behavior and data authority.

## Canonical values

The machine-readable source is generated by `DSA\Design\Seam_Vocabulary_Schema::contract()`.

### Roles

`section`, `container`, `hero`, `lead`, `eyebrow`, `label`, `caption`, `hint`, `micro`, `card`, `media`, `avatar`, `button`, `badge`, `chip`, `nav`, `actions`, `form`, `field`, `input`, `textarea`, `select`, `modal`, `toast`, `testimonial`, `price`, `progress`, `skeleton`, `footer`, `aside`.

Use `aside` for the content role. `sidebar` remains a flow primitive because the old Seam CSS used `data-flow="sidebar"` for the layout pattern.

Roles are semantic/headless by default: `data-role="card"` and `.seam-card` describe meaning for CSS, JavaScript, audits, Bricks, and AI tools, but they do not paint a generic card box, button, modal, badge, field, background, border, shadow, padding, or radius. Build appearance with neutral Seam primitives, universal Kiwe/Seam tokens, and site CSS/classes. Prefer the Seam Class Vocabulary first, such as `seam-horizontal-rail`, `seam-vertical-rail`, `seam-feature`, `seam-story`, `seam-card`, `seam-size-xl`, and `seam-density-spacious`, instead of project-locked names.

### Flows

`stack`, `row`, `cluster`, `inline`, `grid`, `dense`, `sidebar`, `center`, `spread`, `cover`, `frame`, `reel`, `horizontal-rail`, `vertical-rail`.

### Scenes

`dramatic`, `elevated`, `standard`, `compact`, `micro`.

### Tones

`brand`, `accent`, `neutral`, `muted`, `success`, `warning`, `danger`, `info`, `surface`, `inverse`.

### States

`loading`, `disabled`, `selected`, `current`, `error`, `success`, `warning`, `collapsed`, `featured`, `hidden`, `print-hidden`.

`data-state` is space-separated:

```html
<article data-role="card" data-state="featured loading">
```

### Motion

`fade-up`, `scale-in`, `view-fade-up`.

### Global identity classes

Set these on `body` only when a site intentionally opts into a Seam-wide identity:

- `seam-heading-balanced`, `seam-heading-quiet`, `seam-heading-bold`
- `seam-motion-minimal`, `seam-motion-standard`, `seam-motion-expressive`

Seam no longer ships global body shape, density, or elevation modes because those implied hidden appearance. Shape is now an explicit local utility only: `data-shape="square"`, `data-shape="sharp"`, `data-shape="soft"`, `data-shape="rounded"`, `data-shape="pill"`, `data-shape="circle"`, or the matching `.seam-shape-*` classes.

## Behavior attributes

The old standalone Seam runtime had data-binding attributes such as `data-seam-bind` and `data-seam-each`. Those names are reserved, but marketplace themes and generated page sections must not use them to create cart, checkout, auth, PhoneKey, service-worker, browser-history, or Bricks-query authority.

Behavior revival is a later opt-in runtime batch. Batch 1 is presentation-only.

## Kiwe DSA shadow attributes

Normal pages and Bricks layouts should use public Seam vocabulary:

```html
<section data-role="section" data-flow="stack">
```

Kiwe DSA panels use protected shadow metadata instead:

```html
<section class="dsa-panel" data-seam-root="kiwe-dsa" data-seam-role="modal" data-seam-flow="stack">
```

This is intentional. Public Seam role selectors are semantic/headless, but live DSA sheets/screens still have their own strict geometry and theme CSS, so the runtime adds `data-seam-*` metadata after render. Tools, tests, and AI reviewers can identify roles such as cart item cards, FBT rails, forms, CTAs, media, badges, prices, and action rows without allowing site CSS to restyle the AppShell.

DSA may also adopt public Seam classes when the class is proven safe inside the AppShell. The first approved live-adoption set is intentionally narrow: `seam-eyebrow`, `seam-caption`, `seam-price`, and tone classes on existing DSA text/price landmarks. Do not assume that every protected DSA landmark should also receive its public class. Cards, buttons, inputs, media, badges, dock geometry, checkout, and sheet layout stay protected-only primarily for AppShell isolation from site CSS.

The machine-readable `appShellAdoption` map in `seam-vocabulary.json` is the decision source:

- `public-adopted` means Kiwe core may add the public `seam-*` class to live DSA markup.
- `shadow-only` means Kiwe core should keep the protected `data-seam-*` metadata and avoid the public class inside sheets/screens.
- `authority-only` means Seam and themes must not create behavior or state authority for that concern.

Runtime helpers expose the same map:

```js
window.Seam.adoption('button') // { level: 'shadow-only', role: 'button', reason: '...' }
window.Seam.adoption('price')  // { level: 'public-adopted', role: 'price', classes: [...] }
window.DSA.ui.seam.adoption('checkout') // { level: 'authority-only', role: 'checkout' }
```

Use the audit command before proposing a promotion from `shadow-only` to `public-adopted`:

```bash
node tools/ui-theme/audit-seam-adoption.cjs
```

The audit is intentionally conservative. A role should move toward public adoption only when the public Seam class no longer carries AppShell-breaking layout, spacing, shape, color, motion, media, or geometry declarations for that DSA context.

Do not add `data-seam-*` manually in marketplace themes or Bricks page sections. Use the public vocabulary unless you are working inside Kiwe core runtime code.
